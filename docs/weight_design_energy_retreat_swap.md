# 重み設計：エネルギー付与先・にげる・いれかえ

機械学習で重みを付ける対象として、以下 3 種類の「選択」を扱う設計メモ。
実装は行わず、重みの軸・式・差し込み位置のみ整理する。

---

## 1. エネルギー付与先

### 1.1 現状ロジック（`turn.py` の `_try_attach_energy_auto`）

- **選択肢**: バトル場（active）またはベンチの各枠（bench_index）。
- **スコア**: そのポケモンに 1 つ付けたときに相手に与えられる **有效ダメージ** `dmg`。
- **同点時**: バトル場を優先（`x[0] is None` で True 側を選ぶ）。
- **除外**: すでに技の必要エネルギーに達しているベンチは候補から外す。また「バトル場が 0 個」「進化のために必須」のときは付与先を選ばず固定。

### 1.2 重みの置き方（案）

- **軸**: 「どのポケモン（カード）に付けるか」に対するバイアス。
  - 候補は「バトル場のカード」または「ベンチ i のカード」なので、**カード識別子**（例: `card.id` や `(name, regulation)`）ごとに 1 つ重みを持つ。
- **式（案）**:
  - 現状: `score = dmg`（有效ダメージのみ）。
  - 重み付き: `score = dmg + w_energy_attach[card_id]` または `score = dmg * (1 + w_energy_attach[card_id])`。
  - 未学習のカードは `0` やデフォルト値にして、既存ロジックに寄せる。
- **差し込み位置**:
  - `candidates` を作ったあと、各 `(target, dmg)` に対応する **カード** を取得し、その `card_id` で `w_energy_attach` を参照してスコアを補正。
  - `best = max(candidates, key=...)` の key を「補正後スコア」に変更。同点時は現状どおり「バトル場優先」を維持するとよい。

### 1.3 学習で使えそうなシグナル

- そのターンにエネルギーを付けたポケモンが、のちに **KO を取った** / **試合に勝った** なら正の報酬。
- 「付けたがすぐきぜつされた」なら負の報酬、など。ログから「付与先 card_id」と「試合結果・KO 貢献」を紐付ける。

## 2. にげる（逃げ先の選択）

### 2.1 現状ロジック（`turn.py` の `run_turn_auto` 内）

- **条件**: `would_be_koed and not can_ko_opponent and retreat_helps and` エネルギーコスト支払い可能、のときだけにげるを検討。
- **選択肢**: ベンチのインデックス（誰と入れ替えるか）。
- **スコア**（1 段階目）: 各ベンチ `i` に対して  
  `(can_ko, dmg, hp)`  
  - `can_ko`: そのベンチで相手バトル場を KO できるか 0/1。  
  - `dmg`: そのベンチの有效ダメージ。  
  - `hp`: そのベンチの HP。  
  → タプルで最大を 1 体選ぶ。
- **フォールバック**: 誰も「スコア」で選ばれないときは、  
  - 相手の有效ダメージから **生存できる** ベンチの中では `(hp, attached_energy)` 最大、  
  - それもいなければ全ベンチで `(hp, attached_energy)` 最大。

### 2.2 重みの置き方（案）

- **軸**:
  - **A) カードごと**: 「このポケモン（card_id）を逃げ先として選ぶこと」へのバイアス。  
    `score = (can_ko, dmg, hp)` のあと、またはスカラー化した値に `w_retreat_target[card_id]` を加算。
  - **B) スコア要素の重み**: `can_ko` / `dmg` / `hp` の重要度を係数で変える（例: `a*can_ko + b*dmg + c*hp`）。  
    こちらは「状況ごとの好み」を学習しやすいが、既存のタプル順序と両立させるなら、スカラー化してからカードバイアスを足す形が扱いやすい。
- **式（案）**:
  - スカラー化: `raw = can_ko * C_ko + dmg * C_dmg + hp * C_hp`（C_* はスケール係数。既存の優先順位が再現できるようにする）。
  - 重み付き: `score = raw + w_retreat_target[card_id]`。  
  フォールバック（生存優先・HP 優先）の部分も、同じく「カードバイアス」を足すだけで一貫できる。
- **差し込み位置**:
  - ベンチループ内で `(can_ko, dmg, hp)` を計算したあと、`bp.card` の `card_id` で `w_retreat_target` を参照し、スカラー化した値に加算してから `best_idx` を決定。
  - フォールバックの `max(..., key=lambda x: (x[1], x[2]))` なども、key に `+ w_retreat_target[card_id]` を入れるか、別関数で「逃げ先スコア」を返すようにするとよい。

### 2.3 学習で使えそうなシグナル

- 逃げ先に選んだポケモンが **次のターンで相手を KO した** / **試合に勝った** → 正。
- 逃げ先が **次のターンでやられた** / **何もできずに負けた** → 負。  
ログから「にげるで出した card_id」と「その後の結果」を紐付ける。

## 3. いれかえ（グッズでバトル場を入れ替える先）

### 3.1 現状ロジック（`turn.py` の `run_turn_auto` 内）

- **条件**: `would_be_koed and not can_ko_opponent` かつ、いれかえグッズを手札に持っているとき。
- **選択肢**: ベンチのインデックス（誰をバトル場に出すか）。
- **スコア**:
  - 相手の有效ダメージから **生存できる** ベンチがいる → その中で **HP 最大** の 1 体。
  - いない → 全ベンチで **HP 最大** の 1 体。

### 3.2 重みの置き方（案）

- **軸**: 「いれかえでバトル場に出すポケモン（card_id）」へのバイアス。
- **式（案）**:  
  - 現状は `(生存できるか, hp)` で実質 1 段階。  
  - スコアをスカラー化: 例 `score = (1 if survives else 0) * BIG + hp`。  
  - 重み付き: `score += w_swap_target[card_id]`。  
  生存を最優先したければ、`w_swap_target` は小さめの範囲にしておく。
- **差し込み位置**:
  - `survives` を計算したあと、`best_bench` を決めるところで、各ベンチの `(survives, hp)` に `w_swap_target[card_id]` を足したスコアで `max` を取る。

### 3.3 学習で使えそうなシグナル

- いれかえで出したポケモンが **生存して KO を取った** / **勝った** → 正。
- 出したが **すぐ KO された** / **役に立たず負けた** → 負。

## 4. きぜつ時の繰り出し（`_promote_from_bench`）

### 4.1 現状ロジック（`state.py`）

- **条件**: 自分のバトル場が空になったとき、ベンチから 1 体を選んでバトル場に出す。
- **スコア**:
  - 次の自分のターンで **相手バトル場を KO できる** ベンチがいる → その中で **HP 最大**。
  - いない → 全ベンチで **HP 最大**。

### 4.2 重みの置き方（案）

- **軸**: 「繰り出しでバトル場に出すポケモン（card_id）」へのバイアス。
- **式（案）**:  
  - 現状は「can_ko なら can_ko 内で HP 最大、否则 HP 最大」。  
  - スカラー化: `score = (1 if can_ko else 0) * BIG + hp`。  
  - 重み付き: `score += w_promote[card_id]`。
- **差し込み位置**:  
  - `_promote_from_bench` 内で、候補インデックスごとに `can_ko` と `hp` からスコアを計算し、`w_promote[card_id]` を足してから `best_idx` を決定。

### 4.3 学習で使えそうなシグナル

- 繰り出したポケモンが **次のターンで相手を KO した** / **試合に貢献して勝った** → 正。
- 繰り出したが **何もできずにやられた** → 負。

---

## 5. 重みの共通化・データ構造

- **エネルギー付与**: `w_energy_attach[card_id]`（付与先のカード）。
- **にげる先**: `w_retreat_target[card_id]`（逃げ先のカード）。
- **いれかえ先**: `w_swap_target[card_id]`（いれかえで出すカード）。
- **繰り出し**: `w_promote[card_id]`（きぜつ時に繰り出すカード）。

「どのポケモンを前に出すか」は 3 種類（にげる・いれかえ・繰り出し）あるが、  
- いれかえと繰り出しは「誰をバトル場に出すか」で同じなので、`w_swap_target` と `w_promote` を 1 つにまとめて `w_bench_to_active[card_id]` にする、  
- にげるは「逃げ先として選ぶ」なので、攻撃面（can_ko, dmg）の情報が効いている点が違う。  
必要に応じて「逃げ先専用」と「繰り出し・いれかえ共通」で分けておくと、学習時に解釈しやすい。

---

## 6. 実装時の注意（共通）

- 重みは「いないキーは 0」として扱い、**重みなし＝現状ルール** に戻せるようにする。
- スカラー化するとき、`(can_ko, dmg, hp)` などの既存の優先順位が崩れないよう、係数（BIG, C_ko, C_dmg など）は既存の辞書順と整合させる。
- 重みの参照は **GameState や PlayerState に持たせる** か、**モジュールレベルの dict** を引数で渡すか、どちらかに揃えるとよい（後で学習ループから更新しやすい方で決める）。
